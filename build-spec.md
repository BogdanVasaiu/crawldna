# docdna — Build Specification

> **For the AI building this:** This is the complete spec for a general, task-driven web crawler. Build it exactly to the **Core Principles** and the interface **contracts** below. Where a detail isn't specified, choose the simplest option that respects the principles. The name `docdna` is a placeholder — keep it consistent but it will be renamed.

---

## 1. What you are building

A **general, task-driven web crawler**. The user gives it one or more website links, each with a **task** — a natural-language instruction describing what to extract. It crawls each site to fulfil its task and outputs clean Markdown.

**Its defining capability:** on each page it can **take actions to reveal content that only appears after interaction** — clicking tabs, expanding accordions, pressing "load more", scrolling for lazy content — i.e. content a plain fetch never sees. After revealing everything, it **extracts what's relevant to the task**, then **follows the other useful links it finds**, like a crawler.

It runs three ways from a **single core**:

1. **CLI** — point it at link(s) + task(s), get Markdown.
2. **Optional Web UI** — a control panel to set links/tasks, run, and watch live.
3. **Importable library** — another Node project (`refdna`) imports it and consumes its results in production.

**Tasks are arbitrary.** Examples:
- `"Extract all documentation"` — the task refdna needs, run across many docs sites (Firebase, Vuetify, …).
- Any other extraction task, on documentation **or** non-documentation websites.

A task can be **shared** across all links, or set **per link**.

**Precision matters — especially for the documentation task.** The output is consumed by programmers and AI assistants, so completeness (nothing missing) is more important than speed. Slower is acceptable. Never silently skip content; emit a warning when completeness can't be guaranteed.

---

## 2. Core Principles (non-negotiable)

- **General and task-driven.** The engine is generic; specific tasks can be sped up with *profiles* (§4), but the crawler must work for any task on any site.
- **Node.js + plain ESM JavaScript (`.mjs`).** No TypeScript, no build/compile step. Files run directly with `node`.
- **Minimal, bounded dependencies.** Lean on Node built-ins (`fetch`, `node:util.parseArgs`, `node:http`, `node:events`, web streams). Only the small set in §10 is allowed in core. Anything heavy must be optional.
- **One headless core. Thin faces.** All logic lives in a core library that **emits an event stream** and **resolves to a result**. The CLI, UI, and refdna are only *consumers* of that core. No crawling/strategy logic in the CLI or UI.
- **The browser is the actuator for actions, and an optional, lazy-loaded dependency.** Taking actions / revealing hidden content requires a real browser (Playwright), so crawls that need actions require it installed once (`npx playwright install chromium`). Pure structured/static extraction (e.g. a docs site exposing `llms-full.txt`) runs **without** it.
- **Easy install.** Standalone: `git clone` → `npm install` → `node bin/cli.mjs <url>`. Library: `npm install docdna` → `import`.
- **MIT licensed.**

## 3. The engine — how it crawls a page (the general behaviour)

This is the heart of the tool and the default for any task/site. For each page, given the task:

1. **Load** the page (plain `fetch`; escalate to the browser when actions or JS rendering are needed).
2. **Reveal hidden content.** Take actions to surface content generated by interaction — expand tabs/accordions, click "load more"/"show more", scroll for lazy content. An Ollama model decides what is worth interacting with, guided by the task. Element discovery is *behaviour-aware* (not just by HTML tag) — see §11.3.
3. **Extract** the content relevant to the task, as clean Markdown.
4. **Discover and follow useful links.** Surface *every* destination on the page (in-content, nav/footer/app-bar, JS/JSON route tables) **exactly as written, with no URL-shape assumptions**, and let the **AI decide** which are real, on-task pages — recognising site-specific routing/pagination (fragment routes like `#/contact`, query routes like `?view=pricing`, bespoke schemes) while skipping same-page anchors. Queue the chosen ones and continue like a crawler — deduplicating by content and staying in scope — until there is nothing left to do. The algorithm stays generic; the intelligence is in the AI, so new routing styles need no code changes.

**Guardrails (these enable autonomy safely; they are not per-site logic):** a max number of actions per page, a max number of pages, state-fingerprint loop detection (don't repeat a state already seen), and a retry on a malformed model action.

## 4. Task profiles (accelerators)

A task may have a **profile** that adds faster, more reliable paths than open-ended exploration. The engine in §3 is always the fallback. Build this profile first:

### Documentation profile
Applies when the task is `"Extract all documentation"` (or the site is detected as documentation). Because docs sites are highly regular and refdna needs **completeness**, prefer authoritative *complete* sources over exploration. Try in order, stop at the first that yields the full page set:

- **`/llms-full.txt`** — if present, it already contains the entire documentation as clean Markdown in one file. Split by its sections; done. (Read `/llms.txt` too if present, but treat it as a *curated, partial* index, never as the complete list.)
- **`/sitemap.xml` + navigation tree** — follow sitemap indexes, collect every URL, filter to the docs section; also read the rendered sidebar/nav tree as a second source of the full list. Fetch and extract each page.
- **Framework-aware extraction** — detect the docs generator (§11.2) and use its known content container + nav to enumerate and extract all pages. Per-framework, not per-site.
- **Gaps → the engine (§3).** For pages with interaction-hidden content, and for irregular sites with no structured sources, fall back to the agentic engine. **Emit a `warn` for these** — completeness is not guaranteed.

For any **non-documentation task with no profile**, the crawler uses the engine (§3) end to end. (Other profiles can be added later the same way.)

## 5. Public API (the library contract)

The single most important contract — refdna depends on it. Implement exactly.

```js
import { crawlDocs } from 'docdna';   // will be renamed

const run = crawlDocs(targets, options);

// (a) consume live events
for await (const ev of run) {
  // ev = { type, ...payload }  (see §6)
}

// (b) or get the final result
const result = await run.result;

// (c) control (used by the UI / long jobs)
run.stop();   // graceful stop; result still resolves with what was gathered
```

`run` is **async-iterable** (yields events) **and** exposes `run.result` (`Promise<Result>`) and `run.stop()`.

### Targets — supports per-link tasks

```
targets:
    string                                    // one URL, uses options.task
  | string[]                                  // many URLs, all use options.task
  | { url: string, task?: string }            // one target with its own task
  | Array<{ url: string, task?: string }>     // many targets, each with its own task
```

### Options

```js
{
  task: string,        // shared/default task. Default: "Extract the complete documentation."
  model: string,       // Ollama model for the engine's decisions. Default: "qwen3".
  browser: 'never' | 'auto' | 'always',  // default "auto" — lazy-load Playwright only when needed
  concurrency: number, // parallel page fetches. Default: 4.
  maxPages: number,    // safety cap. Default: 0 (unlimited).
  maxActions: number,  // per-page action cap for the engine. Default: 15.
  include: string | RegExp,  // only crawl URLs matching this
  exclude: string | RegExp,  // skip URLs matching this
  cacheDir: string,    // override the runs-cache root (default <project>/.docdna/runs)
  onEvent: (ev) => void,
}
```

Every run is saved to the runs cache regardless of options; there is no `output`
option (see §9).

### Result

```js
{
  pages: [
    { url, task, title, markdown, meta: { strategy, framework?, fetchedAt, bytes } }
  ],
  stats: { pages, durationMs, strategyCounts: { 'docs:llms-full': n, 'docs:sitemap': n, 'docs:framework': n, agent: n } },
  warnings: [ { url?, reason, message } ],
  run: { id, dir, files: [ { filename, bytes } ] }
}
```

## 6. Event model

The core emits these (`{ type, ...payload }`). The CLI and UI render them.

| type | payload | when |
|---|---|---|
| `site` | `{ url, task }` | starting a site |
| `strategy` | `{ url, strategy, framework? }` | profiler/profile picked a path (`docs:llms-full` / `docs:sitemap` / `docs:framework` / `agent`) |
| `discover` | `{ url, count }` | the page set for a site was enumerated |
| `page` | `{ url, status }` | starting a page |
| `action` | `{ url, action, detail }` | engine action taken (`click` / `scroll` / `expand` / `follow` / `done`) |
| `extracted` | `{ url, title, bytes, preview }` | a page was extracted (preview ≈ first 200 chars) |
| `progress` | `{ done, total }` | overall progress |
| `saved` | `{ runId, dir, files: [{ filename, bytes }] }` | the run was written to the cache |
| `warn` | `{ url?, reason, message }` | completeness / quality / browser-needed warning |
| `error` | `{ url?, message }` | recoverable error |
| `done` | `{ stats, run }` | finished (`run` = `{ id, dir, files }`) |

## 7. CLI

Uses `node:util.parseArgs`. Flags map to the options in §5.

```
docdna <url> [--task "..."]                       # crawl one site
docdna crawl <url> [--task "..."] [--model qwen3-coder:30b]
                    [--browser auto|never|always] [--concurrency 4]
                    [--include "..."] [--exclude "..."] [--max-pages 0]
                    [--cache-dir <dir>]
docdna runs [list|rm <id…>|clear|path]            # manage cached runs
docdna serve [--port 4000]                        # start the Web UI
docdna --help
```

**Per-link tasks:** support either repeated pairs `--url A --task "..." --url B --task "..."`, or a `--targets targets.json` file whose contents are the `targets` array from §5. The CLI prints concise live lines (strategy, actions, extractions, warnings) and a final summary including the saved run id/path. **`docdna runs`** lists, deletes (`rm <id>`), or clears cached runs and prints the cache path.

## 8. Web UI (optional) — control surface + real-time view

Started with `docdna serve`. It must let the user do **everything the CLI does**.

### Server
A tiny `node:http` server, no framework:
- `GET /` → serves `ui/index.html`.
- `POST /start` → body = `targets` + `options` (§5). Starts a crawl via the core.
- `GET /events` → **Server-Sent Events** stream of the core's events (§6).
- `POST /stop` → `run.stop()`.
- `GET /runs` → list cached run summaries; `DELETE /runs` → delete all.
- `GET /runs/:id` → a run's `manifest.json`; `DELETE /runs/:id` → delete it.
- `GET /runs/:id/file?name=…` → one output file's raw Markdown.

One active crawl at a time is fine for v1.

### Page (`ui/index.html`) — two steps
Single page, **vanilla JS** (`EventSource` for the live stream). `preact` + `htm` over ESM import allowed if helpful, not required.

**Step 1 — setup + history:**
- A way to add **multiple links, each with its own task** (a repeatable row: URL + task), plus an option to apply one shared task to all.
- Option controls: model, browser mode, concurrency, include/exclude.
- **Start.**
- A **Previous runs** list (from `GET /runs`): open a run (→ step 2, view mode), delete a single run, or delete all.

**Step 2 — run / view:**
- **Live crawl** (after Start): the real-time view —
  - *Where it's looking* — the site/page currently being processed, with strategy + progress.
  - *The extraction* — extracted pages appearing live, with title.
  - *What it did* — the engine's actions as they happen, warnings highlighted.
  - When the run finishes, the produced **files** appear and each can be read.
  - **Stop** is available while running.
- **Saved run** (opened from history): browse the run's files and read each one.
- **← runs** returns to step 1.

**Do not spend time on styling now** — function over form; visual polish comes at the end. Keep the markup simple so it's easy to restyle later.

## 9. Output format

- Always available via `run.result` (§5).
- **Every run is saved to a cache automatically** (no `output` option). The cache
  root is `<project>/.docdna/runs` by default, overridable with the `cacheDir`
  option / `--cache-dir` flag / `DOCDNA_CACHE_DIR` env var. Each run is one folder
  `<runId>/` containing:
  - **AI-grouped Markdown.** By default the whole extraction is **one `.md`**. The
    model splits it into **several named files only when the task asks** to:
    by **category** (e.g. *"…drinks and pizzas separately"* → `drinks.md` +
    `pizzas.md`), or by **page** (e.g. *"…by pages"* / "per page" → one file per
    crawled page, named from the page: `home.md`, `about.md`, …). The model only
    chooses the layout and the filenames — text stays **verbatim**. For category
    splits any unassigned section is folded into the first file; per-page splits
    are materialised deterministically (group by source page), so output is
    always complete. Each file opens with a short YAML front-matter block
    (`task`, `generatedAt`, `sources`).
  - A **`manifest.json`**: `runId`, `createdAt`, `targets`, `options`, a `files`
    list (`{ filename, title, bytes, pages }`), every page `{ url, task, title,
    strategy, file }`, plus `stats` and `warnings`. **refdna reads this manifest**
    — keep it stable and machine-friendly.
  - A small **`run.json`** summary used to list runs quickly.
- Runs are kept until explicitly deleted (UI button or `docdna runs rm/clear`).

Markdown must preserve headings, fenced code blocks (with language), inline code, tables, and links. Strip site chrome (nav, sidebar, footer, cookie banners).

## 10. Dependencies

**Core (required, small, pure-JS):**
- HTML parsing with CSS selectors — `node-html-parser`
- HTML → Markdown — `turndown` (+ `turndown-plugin-gfm` for tables), fenced code blocks
- XML / sitemap parsing — `fast-xml-parser`
- Ollama client — `ollama`

**External requirement:** a running [Ollama](https://ollama.com) with a tool-calling-capable model (default `qwen3`; `gemma` family also fine).

**Optional (in `optionalDependencies`, lazy-imported):**
- `playwright` — the browser, **the actuator for actions and the engine**. Required for any crawl that takes actions / reveals hidden content; not needed for pure structured/static extraction. User installs the browser once: `npx playwright install chromium`.

**Optional (UI only, if using components):** `preact`, `htm`.

No native-compilation deps. No bundler. No TypeScript.

## 11. Implementation notes

### 11.1 Core / event bus & orchestration
`index.mjs` exposes `crawlDocs(targets, options)`. Normalise `targets` into a list of `{ url, task }`. Internally use `node:events` `EventEmitter` (or an async generator). The `run` object exposes `[Symbol.asyncIterator]`, `.result`, and `.stop()`. `stop()` sets a flag; in-flight pages finish, the queue drains, `result` resolves with what was gathered, a final `done` is emitted. For each target, pick a profile (§4) or use the engine (§3); maintain a per-site frontier (queue) with dedupe + scope.

### 11.2 Profiler & docs-framework detection
Detect the generator from page HTML: `<meta name="generator">`, plus characteristic selectors/assets — **Docusaurus** (`.theme-doc-markdown`), **VitePress** (`.vp-doc`, `.VPContent`), **MkDocs/Material** (`.md-content article`, `data-md-*`), **Nextra** (`.nextra-content`), **GitBook**, **Mintlify** (prefer Tier 1, these usually have llms-full.txt), **Sphinx/ReadTheDocs** (`.document`, `role="main"`), **Starlight**, **Fumadocs**, etc. Each maps to a content-container selector and a nav-tree selector. No match → generic readability-style main-content extraction + rely on sitemap for the list. Neither sitemap nor nav → fall back to the engine.

### 11.3 The engine (general per-page loop)
A perceive → decide → act loop driven by Ollama tool-calling, with lazy Playwright as the actuator.
- **Perception:** a compact, *indexed* view of the page — cleaned main text (truncated, ~3000 chars) plus a list of interactive elements, each with a stable id, role, visible label, href. Discover interactivity behaviour-aware: accessibility roles + (where available) real event listeners + heuristics (`cursor:pointer`, `tabindex`). Compute a **state fingerprint** for loop detection.
- **Action space (given to Ollama as tools):** `click(id)`, `scroll()`, `expand(id)`, `extract(reason)`, `follow_links(ids)`, `done(reason)`. (`type(id, text)`, `hover(id)` optional.)
- **Loop:** thread the page's `task` into the system prompt. Each step: send perception → model picks one tool → execute via Playwright → re-perceive → repeat, until `done` or `maxActions`.
- Emit `action` events for the UI; on irregular/interaction-hidden cases under the docs profile, emit `warn`.

### 11.4 Extractor (HTML → Markdown)
Given HTML + a content selector (framework-specific or generic): isolate the main content node, drop nav/aside/footer/script/style, convert with Turndown (`codeBlockStyle: 'fenced'`, GFM tables). Return `{ title, markdown }`.

### 11.5 Concurrency & scope
Fetch pages with a simple concurrency pool (size = `concurrency`). The engine runs sequentially per site (it is stateful). Normalise URLs (strip fragments/trailing slashes), dedupe, and stay within the site's domain unless `include`/`exclude` override.

## 12. Acceptance criteria (definition of done)

- [ ] **Library:** `import { crawlDocs } from 'docdna'` returns a `run` that is async-iterable and has `.result` and `.stop()`. Accepts per-link tasks (§5 `targets`).
- [ ] **Standalone:** `git clone … && npm install && node bin/cli.mjs <url> --task "..."` produces clean Markdown.
- [ ] **General engine:** given a non-documentation site and a task, it reveals interaction-hidden content (clicks/expands), extracts task-relevant Markdown, and follows useful links.
- [ ] **Documentation profile / completeness:** for a Docusaurus site and a sitemap-based site, *every* doc page appears in the output. For a site with `llms-full.txt`, that file is used and the output matches it.
- [ ] **Browser optional:** static/structured crawls run with Playwright *not installed*. Action-taking crawls require it; when it's needed but missing, a clear `warn` instructs the user, and the rest still completes.
- [ ] **UI:** `docdna serve` opens a page where the user can add multiple links each with a task (or a shared task), set options, **Start** and **Stop**, and watch live: the page currently being processed, the extractions, and the actions taken.
- [ ] **Precision:** any site/page where completeness can't be guaranteed produces a `warn` event and a `warnings` entry in the manifest.
- [ ] **Output:** every run is saved to the cache as AI-grouped `.md` (one file by default, several when the task asks to split) plus a stable `manifest.json` and `run.json`. Runs are listable/deletable via `docdna runs` and the UI.
- [ ] **Deps:** limited to §10; `playwright` in `optionalDependencies`; no build step; ESM only.

## 13. Suggested build order

1. **Core skeleton** — `run` object (event bus + `.result` + `.stop()`), `crawlDocs()` + target normalisation, and the HTML→Markdown extractor (§11.4).
2. **Documentation profile, Tier 1 + 2** (llms-full.txt, then sitemap) — gives complete docs crawls with no browser; immediately useful for refdna.
3. **The general engine** (§3, §11.3) — perceive → reveal hidden → extract → follow, with lazy Playwright. The defining capability.
4. **CLI** (`bin/cli.mjs`), including per-link tasks.
5. **UI** (`serve` + `ui/index.html`) — controls + real-time view (minimal styling).
6. **Documentation profile, Tier 3** (framework handlers) + gap-filling via the engine + warnings.
7. **Output writer** + `manifest.json` (can land earlier).

Useful after step 2 (for refdna's docs need) and after step 3 (general crawling); steps 4–7 complete it.

---

## 14. Out of scope (skip and warn — never bypass)

- Login-gated / paywalled content (no credentials).
- CAPTCHA / anti-bot challenges.
- Content rendered only as images or `<canvas>` (not extractable as text).

Skip the page and emit a `warn`; never attempt to circumvent protections.
